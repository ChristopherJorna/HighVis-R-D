HighVis R&D · Programme Hub · 2026-05-06

Direct the shot in Unreal. Let AI handle the photoreal.

HighVis is a tool-agnostic infrastructure for restyling low-fidelity Unreal renders into photoreal video, while preserving camera, composition, and motion. We're building it three ways in parallel. Each one is a fork: same architecture, different model stack, different infrastructure trade-off. We then compare the forks on the same scenes with the same questions, so the answer still holds when the models change.

Recent architectural decisions
Small architectural calls that shape the rest of the build. Documented here so the reasoning sticks.
Local · V2V tuning
F-LC-007: Motion and style are separate dials
How much the AI changes the camera move and how much it applies the new look are controlled by different settings. One dial (the noise schedule) controls motion — turn it down and the camera move stays locked to the original. A separate set of dials (guidance strength and text prompts) controls visual style. This means you can lock the motion first, then push the style without breaking the camera work. Found the sweet spot in 15 test runs.
Local · limitation found
F-LC-008: The new look fades over time
The AI uses a single reference image (the restyled first frame) to guide the entire video. The look is strongest at the start and fades to about 60% after the first second, then holds. This is a fundamental limitation of using one reference frame for a whole clip — the further the video gets from that reference, the weaker its influence. Not fixable by adjusting settings.
Local · next architecture
F-LC-009: Refresh the style every few seconds
Since the look fades because there is only one reference image, the fix is to give each segment its own. Split the video into 2–3 second chunks, generate a fresh reference for each chunk's opening frame, restyle each chunk independently, then join them. Every chunk starts with a strong style anchor, so the fade never gets far. Also opens creative options — different looks per chunk if you want them.
Local · method
Binary search for finding the best settings
Instead of testing every value in small steps (0.4, 0.45, 0.5, 0.55…), make big jumps to narrow in on the answer. If 0.4 gives perfect motion but no style, jump straight to 0.9 instead of 0.55. If 0.9 overshoots, try the midpoint. This found the best balance in about 6 runs instead of 15+. Works for any setting where you are searching a range.
